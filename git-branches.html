Parallel arbeiten –
ohne Angst

Branches sind das mächtigste Werkzeug in Git. Du kannst Ideen ausprobieren, experimentieren, und alles wieder zusammenführen – ohne jemals den stabilen Stand zu gefährden.

Was ist ein Branch? Branch erstellen Wechseln Merge Unerwartete Momente Zusammenfassung
01 / Konzept

Was ist ein Branch –
der parallele Strang

Bisher hast du immer auf main gearbeitet – dem Hauptstrang deines Projekts. Ein Branch ist ein paralleler Strang, auf dem du arbeiten kannst ohne `main` zu berühren.

Analogy

Stell dir vor du designst ein Plakat. Du hast die fertige Version – aber du willst eine rote Variante ausprobieren. Statt das Original zu verändern, machst du eine Kopie, experimentierst damit, und entscheidest dann ob du sie nimmst oder verwirfst. Ein Branch ist genau das – aber für Code.

main
start
commit
commit
merge ←
experiment
farbe
commit

Das Wichtigste: main bleibt immer stabil. Du experimentierst auf einem Branch, und erst wenn du zufrieden bist, führst du ihn mit main zusammen.

02 / Befehle

Branch erstellen
und wechseln

Es gibt zwei Wege einen Branch zu erstellen:

# Weg 1: Zwei Befehle
% git branch experiment
→ erstellt den Branch
% git checkout experiment
→ wechselt zu ihm

# Weg 2: Alles in einem (empfohlen)
% git checkout -b experiment
→ erstellt UND wechselt

Das -b steht für "branch" – es ist eine Abkürzung für beide Schritte zusammen.

Wichtig

Du musst im richtigen Ordner sein – immer auf der Ebene des Projekts, nicht in einem Unterordner wie `css`. Prüfe mit pwd wo du bist.

Nach dem Wechsel bestätigt Git:

Switched to branch 'experiment'
# Du bist jetzt auf dem neuen Branch
03 / Magie

Zwischen Branches wechseln –
zwei Welten, eine Datei

Das ist der Moment der alles verändert. Wenn du zwischen Branches wechselst, tauscht Git den Inhalt deiner Dateien automatisch aus – in Millisekunden.

% git checkout experiment
# style.css zeigt: color: #d44040 ← rote Farbe DA

% git checkout main
# style.css zeigt: keine Farbe ← rote Farbe WEG

Es ist nicht eine andere Datei – es ist dieselbe Datei in zwei verschiedenen Zuständen. Git verwaltet das komplett automatisch.

Häufiger Fehler

Änderungen die noch nicht committed sind, nimmt Git beim Branch-Wechsel mit – oder es gibt Konflikte. Immer committen bevor du den Branch wechselst!

Welchen Branch du gerade bist, siehst du immer mit:

% git branch

experiment
* main
# Der * zeigt deinen aktuellen Branch
04 / Zusammenführen

Merge –
das Beste aus beiden Welten

Wenn du mit deinem Experiment zufrieden bist, führst du es mit main zusammen. Das nennt sich Merge.

1
Zuerst zu main wechseln Du mergst immer in den Branch in dem du gerade bist
git checkout main
2
Branch zusammenführen Git nimmt alle Commits aus experiment und fügt sie in main ein
git merge experiment
3
Zu GitHub pushen main ist jetzt aktuell – in die Cloud hochladen
git push
Pull Request

GitHub schlägt dir nach dem Push eines neuen Branches automatisch einen Pull Request vor – das ist der offizielle Weg um Branches zusammenzuführen wenn du im Team arbeitest. Für Solo-Projekte reicht git merge direkt.

05 / Unerwartete Momente

Was dich überraschen kann –
und wie du rauskommst

Beim Merge können zwei unerwartete Dinge passieren. Hier sind sie erklärt:

⚠ Der Vim-Editor öffnet sich

Manchmal öffnet Git automatisch einen Text-Editor namens Vim – ein sehr alter, mächtiger Editor der sich ganz anders verhält als VS Code. Er erscheint wenn Git eine Merge-Nachricht von dir erwartet.


So kommst du raus:


Esc → dann tippe :wq → dann Enter


`:wq` bedeutet "write and quit" – speichern und schließen. Das reicht vollständig.

Divergent Branches Fehler

Wenn Git sagt "divergent branches" beim Pull, bedeutet das: dein lokales main und das GitHub-main haben unterschiedliche Historien. Lösung:

# Fehler: divergent branches
fatal: Need to specify how to reconcile divergent branches.

# Lösung:
% git pull origin main --no-rebase
# Dann Vim schließen mit: Esc → :wq → Enter
% git push
Warum passiert das?

Wenn du lokal und auf GitHub gleichzeitig Commits hast die der andere nicht kennt, entstehen "divergierende" Historien. Git kann das automatisch zusammenführen – aber es braucht deine Erlaubnis via `--no-rebase`.
