Git mit Augen –
VS Code als visuelle Oberfläche

Du hast Git im Terminal kennengelernt. Jetzt siehst du dasselbe nochmal – aber mit einer grafischen Oberfläche. Kein Befehl, kein Problem.

code . Source Control Terminal vs. VS Code git log Publish Branch Nächster Schritt
01 / Befehl

code . –
VS Code direkt öffnen

Mit diesem kleinen Befehl im Terminal öffnest du VS Code direkt in dem Ordner, in dem du gerade arbeitest:

% code .
# Der Punkt = "in diesem Ordner hier"

VS Code öffnet sich und kennt sofort dein Projekt – alle Dateien, und vor allem: es verbindet sich automatisch mit Git. Kein Setup nötig, VS Code spricht Git von Haus aus.

Warum ist das relevant für deinen Workflow?

Später, wenn du mit Claude + MCP arbeitest, wird Claude Code ebenfalls direkt in VS Code integriert sein. Du wirst denselben Editor nutzen – nur mit KI-Unterstützung drin.

02 / Interface

Source Control –
Git wird sichtbar

Das Git-Icon in der linken Seitenleiste ist dein visuelles Terminal. Die grüne Zahl zeigt dir auf einen Blick: wie viele Dateien haben ungespeicherte Änderungen.

Das "M" neben der Datei steht für Modified – verändert. VS Code zeigt dir den Status jeder Datei direkt an. Andere Buchstaben die du sehen wirst:

U = Untracked → neue Datei, Git kennt sie noch nicht
M = Modified → bekannte Datei, wurde verändert
A = Added → gestaged, bereit zum Commit
D = Deleted → Datei wurde gelöscht
03 / Vergleich

Terminal vs. VS Code –
dieselbe Sprache, zwei Dialekte

Alles was du im Terminal tippst, kannst du in VS Code mit Klicks machen. Es ist kein Unterschied für Git – das Ergebnis ist identisch. Du wählst was sich für dich besser anfühlt.

⌨ Terminal
🖱 VS Code
git status Zeigt geänderte Dateien
Source Control öffnen Siehst alle Änderungen visuell
git add README.md Datei stagen
+ Button neben der Datei klicken Datei in den Wartebereich legen
git commit -m "Nachricht" Commit erstellen
Nachricht eintippen → Commit klicken Commit erstellen
git log --oneline Historie anzeigen
Timeline in der Seitenleiste Alle Commits visuell sehen
Empfehlung

Lern beides. Das Terminal gibt dir ein tieferes Verständnis was Git tut. VS Code ist schneller im Alltag. Profis wechseln ständig zwischen beiden.

04 / Befehl

git log –
Der Zeitstrahl deines Projekts

Mit diesem Befehl siehst du die gesamte Geschichte deines Projekts:

% git log --oneline

dae07b1 (HEAD -> main) README Aktualisiert
3c8b826 Erster Commit

Das --oneline macht die Ausgabe kompakt – eine Zeile pro Commit. Ohne es bekommst du auch Datum, Author und die volle Commit-ID.

dae07b1 · HEAD → main
README Aktualisiert
Aktuellster Stand · Du bist hier
3c8b826
Erster Commit
Der Anfang des Projekts · root-commit

HEAD ist ein Zeiger – er zeigt immer auf den Commit, auf dem du gerade arbeitest. In der Regel ist das der neueste. Aber du könntest HEAD auch auf einen älteren Commit zeigen lassen, um in der Zeit zurückzureisen.

05 / Nächster Horizont

Publish Branch –
Der Knopf der auf GitHub wartet

Nachdem du deinen Commit gemacht hast, hat VS Code dir diesen Button gezeigt:

↑ Publish Branch
# "Dein lokales Projekt auf GitHub hochladen"

Das ist die Brücke zwischen lokal und Cloud. Alles was du bis jetzt gemacht hast, lebt nur auf deinem Mac. Mit "Publish Branch" landet es auf GitHub – gesichert, geteilt, zugänglich von überall.

Und genau das ist der nächste Schritt. Wenn das passiert ist, öffnet sich der Weg zu Claude + MCP + Figma.

✓ Git lokal → ✓ VS Code → → GitHub → Claude + MCP → Figma
