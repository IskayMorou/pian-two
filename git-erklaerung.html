01 / Konzept

Git ist wie ein
sehr geduldiger Buchhalter

Stell dir vor, du arbeitest an einem UI-Design in Figma. Du machst täglich Änderungen – mal gefällt dir ein Layout, mal verwirfst du es wieder. Irgendwann fragst du dich: Wie war das nochmal vor drei Wochen?

Genau dafür ist Git gebaut. Es ist ein System, das jede Änderung in deinem Projekt aufzeichnet, benennt und dauerhaft speichert – aber nur dann, wenn du ihm sagst: "Diese Änderung ist wichtig, speichere sie."

Analogy

Git ist wie ein Notizbuch, in dem du zu bestimmten Momenten schreibst: "Stand heute, 14 Uhr – das Projekt sieht so aus." Du kannst jederzeit zu jedem dieser Momente zurückblättern.

Der wichtige Unterschied zu einer normalen "Speichern"-Funktion: Bei Git entscheidest du aktiv, wann ein Moment festgehalten wird und wie er heißt. Das macht es zu einem mächtigen Werkzeug – aber auch zu einem, das ein bisschen Gewöhnung braucht.

02 / Befehl

git init –
Den Buchhalter einstellen

Als du diesen Befehl ausgeführt hast:

% git init
Initialized empty Git repository in /Users/iskay/Documents/your-project/.git/

...hat Git einen unsichtbaren Ordner namens .git in deinem Projektordner erstellt. Dieser Ordner ist das Gehirn von Git – er speichert die gesamte Geschichte deines Projekts.

Wichtig zu wissen

Du siehst diesen .git-Ordner nicht im Finder, weil er versteckt ist (der Punkt am Anfang bedeutet "versteckt" auf Mac). Aber er ist da. Lösche ihn nie – sonst verlierst du die gesamte Git-Historie.

git init machst du nur einmal pro Projekt. Ab diesem Moment ist Git für diesen Ordner zuständig.

03 / Befehl

git status –
Den Überblick behalten

Dieser Befehl ist dein bester Freund. Er zeigt dir jederzeit den aktuellen Zustand deines Projekts aus Git's Perspektive. Du kannst ihn so oft ausführen wie du willst – er verändert nichts, er schaut nur.

% git status

On branch main
# Du bist auf dem "Hauptstrang" deiner Arbeit

No commits yet
# Noch nichts dauerhaft gespeichert

Untracked files:
README.md
# Diese Datei existiert, aber Git ignoriert sie noch

Das Wort "Untracked" bedeutet: Git sieht die Datei, aber es beobachtet sie nicht. Es ist als würdest du einen Assistenten einstellen, ihm aber noch keine Aufgaben geben.

Merke dir

Führe git status regelmäßig aus – immer wenn du nicht sicher bist, wo du stehst. Es ist dein Kompass.

04 / Befehl

git add –
Auswählen was gespeichert wird

Hier wird es interessant. Bei Git speicherst du nicht einfach alles auf einmal – du wählst gezielt aus, welche Änderungen im nächsten "Foto" (Commit) enthalten sein sollen.

Analogy

Stell dir vor, du fotografierst dein Projekt für ein Portfolio. Bevor du auf "Auslösen" drückst, arrangierst du was im Bild sein soll. git add ist das Arrangieren – git commit ist der Klick auf den Auslöser.

% git add README.md
# Nur diese eine Datei wird vorbereitet

% git add .
# Der Punkt bedeutet: ALLE geänderten Dateien vorbereiten

Nach dem git add ist die Datei in der sogenannten "Stage" – einer Art Wartebereich. Sie ist bereit, aber noch nicht dauerhaft gespeichert.

Changes to be committed:
new file: README.md
# "staged" = im Wartebereich, bereit zum Speichern
05 / Befehl

git commit –
Den Moment einfrieren

Der Commit ist der wichtigste Moment in Git. Du speicherst den aktuellen Stand dauerhaft und unwiderruflich in der Geschichte deines Projekts. Mit einer Nachricht, die beschreibt was du getan hast.

% git commit -m "Erster Commit"

[main (root-commit) 3c8b826] Erster Commit
1 file changed, 0 insertions(+), 0 deletions(-)
create mode 100644 README.md

Was hat Git dir hier gesagt?

main
Der aktuelle Branch (Strang) Du arbeitest auf dem Hauptstrang deines Projekts. Später kannst du parallele Stränge erstellen – z.B. um etwas auszuprobieren ohne den Hauptstand zu verändern.
root-commit
Der allererste Commit Jedes Projekt hat genau einen "root-commit" – den Ursprung. Alle zukünftigen Commits bauen darauf auf.
3c8b826
Die eindeutige ID des Commits Jeder Commit bekommt automatisch einen einzigartigen Code. Du kannst damit jederzeit zu genau diesem Moment zurückspringen.
-m "..."
Die Commit-Nachricht Das "-m" steht für "message". Diese Nachricht ist für dich – schreib immer etwas Sinnvolles, das dir in 3 Monaten noch sagt, was du damals geändert hast.
06 / Überblick

Die drei Zustände –
der komplette Kreislauf

Alles was du getan hast, lässt sich in einem einfachen Ablauf zusammenfassen. Jede Datei in deinem Projekt befindet sich immer in einem dieser drei Zustände:

Zustand 1
Untracked
Datei existiert, Git ignoriert sie
git add →
Zustand 2
Staged
Im Wartebereich, bereit zum Speichern
git commit →
Zustand 3
Committed
Dauerhaft in der Geschichte gespeichert

Wenn du später neue Dateien erstellst oder bestehende änderst, beginnt der Kreislauf wieder von vorne. Das ist der grundlegende Rhythmus der Arbeit mit Git.

Gute Commit-Gewohnheiten

Committe oft und mit sinnvollen Nachrichten. Nicht "Änderungen" oder "Update" – sondern "Navigation Farbe auf Blau geändert" oder "Mobile Ansicht für Header fertig". Dein zukünftiges Ich wird es dir danken.

07 / Ausblick

Wo du hinwillst –
der große Plan

Dein Ziel: UI-Design mit Claude, MCP und GitHub

Was du gerade lernst ist die Grundlage für einen modernen, KI-unterstützten Design-Workflow. Hier ist wie das aussehen wird:

Claude (KI) + MCP (Verbindung) + GitHub (Speicher) → Dein UI-Projekt

Git und GitHub sind dabei die Grundlage – das Fundament auf dem alles andere aufbaut. Wenn du Git verstehst, kannst du:

→ Mit Claude zusammenarbeiten, der deinen Code direkt in GitHub committen kann
→ Verschiedene Design-Varianten parallel testen (Branches)
→ Jederzeit zu einer älteren Version zurückgehen
→ Mit anderen Designern oder Entwicklern zusammenarbeiten

Du bist auf dem richtigen Weg. Die ersten Schritte sind immer die schwersten – ab jetzt wird es vertrauter.

08 / Vokabular

Begriffe
in einfachen Worten

Mac · kein Git
Cmd + S Datei speichern auf dem Mac – hat nichts mit Git zu tun
Grundbefehle
git status Status checken was hat sich verändert?
git add Datei stagen für Git vorbereiten
git commit Datei committen in Git-Geschichte festhalten
git log Historie lesen was wurde wann committed?
GitHub · sync
git push Änderungen pushen von Mac zu GitHub hochladen
git pull Änderungen pullen von GitHub zu Mac herunterladen
git fetch Änderungen fetchen nachschauen was neu ist, ohne herunterzuladen
Branches
git branch Branch anzeigen auf welchem Strang bin ich?
git checkout Branch wechseln zu einem anderen Strang wechseln
git merge Branches mergen zwei Stränge zusammenführen
Korrekturen
git restore Änderungen verwerfen Datei auf letzten Commit zurücksetzen
git rm Datei entfernen aus Git-Tracking löschen

Was macht der HEAD?

Der HEAD ist wie ein Lesezeichen in deiner Git-Geschichte. Er zeigt immer auf den aktuellen Commit, an dem du arbeitest. Wenn du einen neuen Commit machst, bewegt sich der HEAD automatisch mit – er zeigt immer auf den neuesten Stand.

Wenn du zu einem älteren Commit zurückspringst (z.B. mit git checkout), bewegt sich der HEAD zu diesem Commit. Das ist wie ein Zeitsprung in deiner Projektgeschichte. Du kannst jederzeit zurück zum neuesten Commit springen, indem du git checkout main (oder den Namen deines Hauptbranches) eingibst.

Es hat ein symbol in VSC. Diesen kleinen Kreis mit dem Punkt in der Mitte (wie eine Zielscheibe). Die fette Schrift: In der Liste links steht der oberste Punkt in fetter Schrift. Das zeigt dir visuell: Das ist der Stand, den du gerade vor Augen hast.
Kurz gesagt: Der HEAD ist dein „Du bist hier“-Aufkleber auf der Landkarte deines Projekts.
