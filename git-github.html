01 / Konzept

Git vs. GitHub –
der wichtigste Unterschied

Das ist eine Verwechslung die fast jeder am Anfang macht. Die beiden Namen klingen ähnlich, sind aber grundlegend verschieden:

Auf deinem Mac
Git
Das Versionskontrollsystem. Läuft lokal, braucht kein Internet.
git push
⟶
In der Cloud
GitHub
Die Plattform. Speichert dein Projekt online, zugänglich überall.

Analogy

Git ist dein Notizbuch auf dem Schreibtisch – du schreibst darin, es bleibt bei dir. GitHub ist die Bibliothek, wo du eine Kopie einreichst. Andere können es einsehen, du kannst es von überall abrufen.

Für deinen späteren Workflow mit Claude + MCP ist GitHub entscheidend: Claude kann direkt auf dein Repository zugreifen, Änderungen lesen, committen und pushen – alles automatisch.

02 / Schritt

Ein neues Repository
auf GitHub erstellen

Ein Repository (kurz: Repo) ist der Container für dein Projekt auf GitHub – mit allen Dateien, der gesamten Historie, und den Einstellungen.

1
github.com → "+" → New repository Oben rechts auf dein Profilbild oder das Plus-Symbol klicken
2
Repository Name vergeben Gleicher Name wie dein lokaler Ordner ist empfehlenswert – z.B. pian-one
3
Visibility: Public lassen Für Lernprojekte ist Public in Ordnung. Private Repos gibt es auch – für sensible Projekte.
4
README, .gitignore, License: ALLE auf Off/No lassen Du hast das alles bereits lokal. Wenn GitHub ebenfalls Dateien erstellt, gibt es einen Konflikt beim ersten Push.
5
Create repository GitHub erstellt ein leeres Repo und zeigt dir die Befehle zum Verbinden
Häufiger Fehler

README beim Erstellen aktivieren obwohl du schon eine lokal hast. Das erzeugt einen Konflikt – GitHub und dein Mac haben dann unterschiedliche Historien und der erste Push schlägt fehl.

03 / Sicherheit

Personal Access Token –
das Terminal-Passwort

GitHub akzeptiert dein normales Passwort nicht im Terminal. Stattdessen brauchst du einen Personal Access Token – ein speziell generiertes Passwort nur für den Terminal-Zugang.

Warum das so ist

Dein normales GitHub-Passwort schützt deinen ganzen Account. Ein Token hat begrenzte Rechte, eine Ablaufzeit, und kann jederzeit widerrufen werden – ohne dein Hauptpasswort zu ändern. Das ist sicherer.

Token erstellen:

1
github.com → Profilbild → Settings Ganz unten im Menü
2
Developer settings → Personal access tokens → Tokens (classic) Ganz unten in der linken Seitenleiste
3
Generate new token (classic) Note: terminal-mac · Expiration: 90 days · Scope: repo anklicken
4
Token sofort kopieren! Er wird nur einmal angezeigt. Speichere ihn in einem Passwort-Manager oder einer sicheren Notiz.
Token sicher aufbewahren

Behandle den Token wie ein Passwort. Teile ihn nie, commit ihn nie in eine Datei, poste ihn nie online. Wenn er verloren geht – einfach einen neuen erstellen und den alten widerrufen.

04 / Befehle

Die drei Befehle –
lokal mit GitHub verbinden

Diese drei Befehle führst du einmalig pro Projekt aus – danach ist die Verbindung dauerhaft gespeichert:

# 1. GitHub als Ziel registrieren ("origin" = Spitzname für GitHub)
% git remote add origin https://github.com/IskayMorou/pian-one.git

# 2. Branch "main" nennen (zur Sicherheit)
% git branch -M main

# 3. Hochladen – erstes Mal mit "-u" um Verbindung zu setzen
% git push -u origin main

Was bedeutet origin? Das ist nur ein Spitzname für die GitHub-URL. Du könntest es auch "github" nennen – aber "origin" ist die universelle Konvention.

Was bedeutet -u? Es setzt die Verbindung dauerhaft. Ab dem zweiten Mal reicht einfach git push – ohne alles andere.

# Was GitHub antwortet wenn es klappt:
Enumerating objects: 6, done.
Counting objects: 100% (6/6), done.
Writing objects: 100% (6/6), 459 bytes | 459.00 KiB/s, done.
* [new branch] main -> main
branch 'main' set up to track 'origin/main'.
🎉

Das hast du erreicht

Dein lokales Projekt ist jetzt auf GitHub. Jeder Commit den du ab jetzt machst, kann mit einem einzigen git push hochgeladen werden.

05 / Workflow

Der tägliche Kreislauf –
wie du ab jetzt arbeitest

Das ist der Rhythmus den du von jetzt an immer wiederholen wirst – bei jedem Projekt, jeden Tag:

Änderung Datei bearbeiten in VS Code
git add . Alle Änderungen stagen
git commit -m Moment festhalten
git push Zu GitHub hochladen

Ab dem zweiten Push reicht wirklich nur noch git push – Git weiß bereits wohin.

Relevanz für Claude + MCP + Figma

Wenn Claude später über MCP auf dein GitHub-Repository zugreift, sieht es genau diese Commits – mit deinen Nachrichten, Zeitstempeln und Änderungen. Claude kann dann selbst committen und pushen, als wäre es ein Mitarbeiter in deinem Team.
